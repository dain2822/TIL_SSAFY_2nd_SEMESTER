- 서킷 브레이커 패턴
    
    : 서비스 간 호출에서 일정 횟수 이상 실패가 발생하면 호출을 차단하여 시스템 전체 장애로 확산되는 것을 방지하는 패턴
    
    : 전기 회로의 차단기처럼 장애가 발생하면 연결을 끊어 시스템을 보호하는 역할
    
    : MSA(Microservice Architecture) 환경에서 필수적인 장애 대응 전략
    
    - 필요한 이유
        - A 서비스 → B 서비스 호출
        - B 서비스가 장애 상태인데 계속 요청하면 A 서비스까지 장애 발생(연쇄 장애)
        
        → 이를 방지하기 위해 일정 조건에서 호출 중단
        
    - 동작 상태
        - Closed(정상 상태)
            
            : 요청 정상 전달
            
        - Open(차단 상태)
            
            : 요청을 아예 보내지 않고 즉시 실패 반환
            
        - Half-Open(시험 상태)
            
            : 일부 요청만 보내서 복구 여부 확인
            
    - 동작 흐름
        1. 정상 상태에서 요청 처리
        2. 실패율이 기준 이상 → Open 상태 전환
        3. 일정 시간 후 Half-Open 상태
        4. 성공하면 Closed, 실패하면 다시 Open
    - 장점
        - 장애 전파 방지
        - 시스템 안정성 향상
        - 빠른 실패 처리(Fail Fast)
        - 서비스 복구 감지 가능
    - 단점
        - 구현 복잡도 증가
        - 잘못된 설정 시 정상 요청 차단 가능
        - 추가 상태 관리 필요
    - 유사 질문
        
        : Circuit Breaker 패턴이 필요한 이유?
        
        : Closed, Open, Half-Open 상태 설명
        
        : Retry와 Circuit Breaker의 차이?
        
        : 장애 전파를 막는 방법?
