- 로드 밸런싱
    
    : 여러 서버에 트래픽을 균등하게 분산시켜 시스템의 성능과 안정성을 높이는 기술
    
    : 하나의 서버에 요청이 집중되는 것을 방지하고, 서비스의 가용성과 확장성을 확보하기 위해 사용됨
    
    : 대규모 웹 서비스나 클라우드 환경에서 필수적인 인프라 기술
    
    - 필요한 이유
        - 사용자 요청이 한 서버에 몰리면 서버 과부하 발생
        - 서버 장애 발생 시 서비스 중단 가능
        
        → 성능 향상, 서버 장애 대응, 확장성 확보
        
    - 로드 밸런싱 방식
        - Round Robin
            
            : 요청을 순서대로 서버에 분배
            
        - Least Connections
            
            : 현재 연결 수가 가장 적은 서버에 요청 전달
            
        - IP Hash
            
            : 클라이언트 IP를 기반으로 동일 서버에 요청 전달
            
        - Weighted Round Robin
            
            : 서버 성능에 따라 가중치를 두고 분배
            
    - 로드 밸런서 위치
        - L4 Load Balancer
            
            : 전송 계층(TCP/UDP) 기반 분산
            
            : 속도가 빠름
            
            : 패킷 정보(IP, Port) 기반 처리
            
        - L7 Load Balancer
            
            : 애플리케이션 계층 기반 분산
            
            : URL, HTTP Header, Cookie 등 기반으로 분배 가능
            
            : 더 정교한 트래픽 제어 가능
            
    - 장점
        - 서버 부하 분산
        - 서비스 가용성 향상
        - 시스템 확장성 확보
        - 장애 대응 가능
    - 단점
        - 추가 인프라 비용 발생
        - 로드 밸런서 자체가 장애 지점(SPOF)이 될 수 있음
        - 설정 및 운영 복잡도 증가
    - 유사 질문
        
        : L4 로드 밸런서와 L7 로드 밸런서의 차이?
        
        : Round Robin과 Least Connection의 차이?
        
        : 로드 밸런서를 사용하는 이유?
        
        : 로드 밸런서가 장애가 나면 어떻게 대응할 수 있는가?
